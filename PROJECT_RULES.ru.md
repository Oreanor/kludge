# Правила проекта — AIR

Краткие и практические правила, чтобы люди и агенты работали согласованно.

## Принципы
- Сначала проектируем типы и контракты, потом сервисы и UI.
- Новая фича: types → service → provider → ui → command.
- Нельзя хардкодить модели, провайдеры или секреты в коде.
- Секреты хранятся в настройках, env или secret storage.
- Ответы моделей проходят через orchestrator, не вызываются из UI напрямую.
- Действия, которые меняют состояние, требуют preview + подтверждение.
- Правки в файлах применяются через патчи/диффы и показываются пользователю.

## Архитектура
- Одна ответственность на модуль.
- `extension.ts` только регистрирует команды, провайдеры и сервисы.
- `services/` — бизнес-логика, оркестрация, интеграции с моделями.
- `webview/` — только UI и передача сообщений (типизированные сообщения).
- `providers/` — адаптеры к LLM, image API, источникам иконок/preview.
- `types/` — общие интерфейсы, события и enum'ы.

## Данные и типы
- Сообщения — discriminated unions.
- Сущности имеют `id`, `createdAt`, `status` там, где нужно.
- Модели запрос/ответ — отдельные типы.
- Без `any`, кроме отмеченных прототипов.
- Файлы передавать как `Uri` + `Range`.

## UI
- Логика UI реактивная.
- Для каждого экрана: loading / error / empty / ready.
- Вкладки восстанавливают состояние после перезагрузки.
- Preview, чат, иконки — отдельные панели.

## Чат и агенты
- Явный контекст: выделение, активный файл, workspace root, доп. файлы.
- Отправлять модели только нужный контекст.
- Поддерживать стриминг ответов при возможности.
- Агент возвращает структурированный результат.
- Человек принимает решение об изменениях.

## Код и патчи
- Генерировать правки как патч и показывать unified diff.
- Проверять, что файл не менялся перед аплаем.
- Соблюдать форматирование и стиль проекта.

## Webview и preview
- Preview отделён от логики редактора.
- Обязательно CSP + nonce в webview.
- Источник preview явный, с fallback для ошибок.

## Модели и ключи
- Провайдеры через адаптеры.
- Запрашивать ключи только по необходимости, хранить безопасно.

## Качество
- Acceptance criteria для каждой фичи.
- Базовые тесты для новых экранов.
